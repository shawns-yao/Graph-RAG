# 三路融合检索

## 简历原文

> 路由根据 query type 选择检索链路（vector / BM25 / graph / hybrid），召回质量不足时自纠循环根据反思结果逐步升级。Hybrid 检索按通道优先级去重合并，并交由 cross-encoder 做最终重排，避免手写权重融合。

---

## 一、为什么需要多路检索？

单一检索方式各有盲区：

| 检索方式 | 擅长 | 盲区 |
|---------|------|------|
| Vector（向量） | 语义相似的内容 | 精确术语、缩写（如"ACEI"语义模糊） |
| BM25（字面） | 精确关键词匹配 | 同义词、换一种说法就找不到 |
| Graph（图遍历） | 实体间关系、多跳推理 | 没有建边的实体对、新实体 |

三路融合的目标：**覆盖语义匹配、字面锚点与多跳关系查询等不同检索场景**。

---

## 二、路由机制：不是一上来就三路并行

### 核心设计：单通道优先，按需升级

路由首先将 query 分类为 5 种类型，映射到对应的**单通道工具**：

| Query Type | 默认工具 | 适用场景 |
|---|---|---|
| simple | vector_search | 简单事实查询（"ACEI 是什么"） |
| relation | cypher_traverse | 关系查询（"ACEI 的禁忌症"） |
| multi_hop | cypher_traverse | 多跳推理（"为什么不用 ACEI 要改用 ARB"） |
| global | comprehensive_search | 全局概览（"列出所有降压药分类"） |
| temporal | temporal_query | 时间相关（"DAPT 疗程多久"） |

### 路由分类方式：确定性规则优先

两级路由级联：

1. **Hard Rule**（确定性规则）：检测关键词/模式，直接选工具
   - 检测到"禁忌"、"副作用"、"替代"→ cypher_traverse
   - 检测到"有哪些"、"列出"、"overview"→ comprehensive_search
   - 短查询（≤4 token）→ vector_search
2. **Pattern**（正则兜底）：多模式计数，哪个类型匹配最多选哪个

大部分查询在 Hard Rule 层就被拦截了，不需要调 LLM。

---

## 三、逐步升级机制

**关键点：不是单通道失败就直接跳到三路并行。**

升级路径是：单通道 A → 单通道 B → 单通道 C → hybrid（三路并行）

具体流程：

1. 路由选 vector_search → 执行
2. 自纠循环评估 → verdict=retry，failure_type="relation_missing"
3. 根据 failure_type 查 retry matrix，选下一个单通道工具（cypher_traverse）
4. cypher_traverse 执行 → 再次评估
5. 如果仍不足 → 才升级到 hybrid_search（三路并行）

**Retry Matrix**（每个工具的升级路径）：

| 当前工具 | 升级顺序 |
|---------|---------|
| vector_search | bm25 → cypher → hybrid → comprehensive |
| bm25_search | vector → cypher → hybrid → comprehensive |
| cypher_traverse | bm25 → vector → hybrid → comprehensive |
| hybrid_search | bm25 → cypher → vector → comprehensive |

升级顺序不是固定的——自纠循环的 reflection 会推荐下一个工具，retry matrix 只是兜底。

---

## 四、Hybrid 融合（hybrid_search 内部）

当升级到 hybrid_search 时，三个通道并行执行：

```
vector_search ──┐
bm25_search  ───┼──→ 通道优先级去重合并 → cross-encoder rerank → top-k
graph_search ───┘
```

### 通道优先级

当前不再使用 RRF，也不再给通道写浮点权重。原因很简单：
vector、BM25、graph 的分数来源不同，强行融合容易制造“看起来科学”的假分数。

融合只做两件事：

- 根据 query type 决定通道顺序
- 按顺序追加结果，已出现的 chunk 去重

默认优先级：

| Query Type | Channel Priority |
|---|---|
| simple | vector → BM25 → graph |
| relation | graph → vector → BM25 |
| multi_hop | graph → vector → BM25 |
| global | vector → BM25 → graph |
| temporal | BM25 → vector → graph |

这样做的含义很明确：relation / multi_hop 先相信图入口，vector 和 BM25 只补充图没有覆盖到的证据。

### 增量通道刷新

如果 hybrid_search 的结果仍不够好，retry 时不需要重跑所有三个通道：

- 自纠循环识别出哪个通道质量差（如 graph 通道返回 0 条结果）
- 只重跑质量差的通道，复用已缓存的好通道结果
- 重新融合

这通过 `channel_cache` 实现：每次检索的结果按通道缓存，下次 hybrid 可以选择性复用。

### Cross-encoder rerank

通道合并后，只用 cross-encoder 做最终排序：

- 输入 `(query, passage)` 对
- 输出相关性排序
- 如果 cross-encoder 不可用，保留通道优先级顺序，不回退到另一套伪分数

这里不再使用 cosine rerank。cosine 只适合召回阶段，最终排序交给 cross-encoder。

---

## 五、并行执行

三个通道通过 ThreadPoolExecutor 并行执行，带超时降级：

- 最大并行数：3（每个通道一个线程）
- 超时：15 秒
- 降级策略：某个通道超时 → 该通道返回空列表，不阻塞其他通道

## 六、面试追问

### Q1：为什么不一开始就跑三路并行？

成本和延迟。三路并行意味着每次查询都要调向量索引 + BM25 索引 + 图遍历，延迟是单通道的 3 倍（即使并行也有开销）。大部分简单查询用 vector_search 一次就够了，没必要浪费资源。

### Q2：为什么不用 RRF？

标准 RRF 本身可以用，但当前项目不需要再引入一个融合分数。通道优先级更透明：relation/multi_hop 就是 graph 优先，simple 就是 vector 优先。真正的最终排序交给 cross-encoder。

### Q3：增量通道刷新具体怎么实现的？

通过 `channel_cache`（一个 dict[query, dict[channel, results]]）。每次检索后，结果按通道名（vector/bm25/graph）缓存。下次 hybrid_search 时，`_hybrid_cache_plan()` 函数检查哪些通道有缓存、哪些需要重跑，把缓存结果作为 `seed_results` 传入 orchestrator，只执行缺失的通道。

### Q4：如果三个通道返回的结果完全不重叠怎么办？

按优先级追加，全部进入候选池，再由 cross-encoder 重排。低优先级通道不会覆盖高优先级通道，但有机会被 cross-encoder 提到前面。

### Q5：rerank 用的是什么模型？

配置项 `RETRIEVAL_RERANKER_MODEL` 指定 cross-encoder。当前代码只保留 cross-encoder rerank，不再保留 cosine / lexical-semantic rerank。

### Q6：vector / BM25 / graph 三个通道的结果数量一样吗？

不一样。默认配置：
- vector：top-10
- bm25：top-10
- graph：top-10（受 max_hops=2 限制）

但实际返回数量取决于索引中有多少匹配。graph 通道如果入口实体没找到，结果会少，后续通道会补充候选。

### Q7：这个方案和 Elasticsearch 的 hybrid search 有什么区别？

ES 的 hybrid search 是 vector + BM25 两路，没有 graph 通道。这个项目多了知识图谱遍历，能处理“A 和 B 是什么关系”这类需要多跳推理的查询。另外当前实现不用 linear combination，也不用 RRF 分数，而是通道优先级合并后交给 cross-encoder 排序。
