# 双节点图谱建模

## 简历原文

> 设计 PhraseNode（实体层）+ PassageNode（段落层）双层结构，MENTIONED_IN 连接实体到原文段落支持证据回溯，RELATED_TO 连接实体到实体支持关系遍历与多跳推理。通过低信息实体过滤与挂载上限控制，降低假边率与遍历失控风险。

---

## 一、为什么需要双节点结构？

传统 Graph RAG 只有实体节点，查到实体后还得回数据库找原文。问题：

1. **证据断链**：图遍历找到了目标实体，但不知道原文在哪，需要额外查询
2. **上下文丢失**：实体节点只存名字和类型，丢失了原文中的完整语境
3. **生成质量差**：LLM 生成答案需要原文段落作为 grounding，只给实体名不够

双节点结构的设计来自 HippoRAG 2（ICML 2025）：**实体层负责图遍历，段落层负责提供证据**。

---

## 二、节点设计

### PhraseNode（实体层）

存储从文档中抽取的实体，是图遍历的主体。

| 属性 | 说明 |
|------|------|
| id | 实体唯一标识（name+type 的 MD5） |
| name | 实体名称（如"ACEI"、"高钾血症"） |
| entity_type | 实体类型（14种：Disease, Drug, DrugClass, Therapy, Test, Biomarker, Anatomy, Procedure, RiskFactor, Pathogen, Population, Device, Threshold） |
| pagerank_score | 来源 chunk 的 PageRank 分数 |
| confidence | 抽取置信度（LLM 抽取的有值，正则注入的为 0） |
| aliases | 别名列表（如 ACEI 的别名：血管紧张素转换酶抑制剂、普利类药物） |
| embedding | 实体名称的向量表示（用于向量检索入口） |

### PassageNode（段落层）

存储原文 chunk 全文，是最终提供给 LLM 生成答案的证据。

| 属性 | 说明 |
|------|------|
| id | chunk 唯一标识 |
| text | chunk 全文（enriched 版本，包含上下文增强） |
| chunk_id | 对应原始 chunk 的 ID |
| embedding | chunk 全文的向量表示（用于向量检索） |

---

## 三、边的设计

### MENTIONED_IN：PhraseNode → PassageNode

**含义**：这个实体在这段原文里被提到了。

```
PhraseNode("ACEI") --MENTIONED_IN--> PassageNode("...ACEI禁用于双侧肾动脉狭窄...")
PhraseNode("ACEI") --MENTIONED_IN--> PassageNode("...ACEI是CKD患者首选降压药...")
PhraseNode("ACEI") --MENTIONED_IN--> PassageNode("...ACEI常见副作用为干咳...")
```

**作用**：查到实体后，沿这条边直接拿到原文证据给 LLM 生成答案。

**建边逻辑**：
1. 对每个实体，收集它的所有表面形式（name + aliases）
2. 扫描所有 chunk，计算匹配分数（全词匹配 > 子串匹配 > 缩写匹配 > 模糊匹配）
3. 按分数降序，取前 3 个 chunk 建边

### RELATED_TO：PhraseNode → PhraseNode

**含义**：这两个实体之间有医疗关系。

```
PhraseNode("ACEI") --RELATED_TO{type:"禁忌"}--> PhraseNode("高钾血症")
PhraseNode("ACEI") --RELATED_TO{type:"替代"}--> PhraseNode("ARB")
PhraseNode("二甲双胍") --RELATED_TO{type:"禁忌"}--> PhraseNode("eGFR<30")
```

**作用**：支持多跳遍历。用户问"ACEI 禁忌什么"→ 沿 RELATED_TO 找到"高钾血症"→ 再沿 MENTIONED_IN 找到包含这个关系的原文段落。

**建边来源**：
1. **LLM 抽取**：骨架 chunk 深度抽取时 LLM 直接输出的关系
2. **句子级模式推断**：同一句话中出现两个实体 + 匹配到关系模式（禁忌/替代/剂量/疗程/副作用/推荐）→ 自动建边

**关系标准化**：建边前通过别名表把端点映射到标准实体名，防止"ACEI"和"血管紧张素转换酶抑制剂"建成两条重复边。

## 四、质量控制机制

### 1. 停用词过滤（实体层面）

```python
_COMMON_ENTITY_NAMES = {
    "patient", "patients", "treatment", "treatments",
    "method", "methods", "result", "results", ...
    "患者", "治疗", "方法", "结果", "研究", "数据", ...
}
```

这些词不会创建 PhraseNode。它们出现频率极高但没有检索价值。

### 2. TF-IDF 低信息实体剔除

如果一个实体的 IDF 值过低（即在大部分文档中都出现），说明它没有区分度，跳过建边。

### 3. 置信度过滤

confidence < 0.7 的实体不建 MENTIONED_IN 边。这些通常是正则注入的低质量实体。

### 4. 挂载上限控制

```python
_MAX_MENTIONED_IN_PER_ENTITY = 3
```

每个实体最多挂 3 个 PassageNode。防止中频实体（如"2型糖尿病"——有意义但出现太多次）挂满整个图，导致遍历时返回过多无关段落。

### 5. 关系类型约束

RELATED_TO 只保留 6 种医疗关系类型：禁忌、替代、疗程、剂量、副作用、推荐。不是任意两个实体共现就连边——必须匹配到关系模式。

### 6. 实体类型约束

建关系时检查目标实体类型是否合理。例如"禁忌"关系的目标必须是 Drug/DrugClass/Therapy/Threshold/Device 之一，不能是 Population 或 Anatomy。

## 五、检索时怎么用？

以查询"ACEI 的禁忌症有哪些"为例：

1. **向量检索入口**：query embedding 与 PhraseNode embedding 做相似度搜索，命中"ACEI"
2. **图遍历**：沿 RELATED_TO（type=禁忌）找到"高钾血症"、"双侧肾动脉狭窄"、"妊娠"
3. **证据回溯**：对每个目标实体，沿 MENTIONED_IN 找到包含该关系的原文段落
4. **组装上下文**：把这些段落拼接后送给 LLM 生成答案

这就是 VectorCypher 的核心模式：**向量入口 → Cypher 遍历 → 上下文组装**。

## 六、面试追问

### Q1：为什么不直接用一种节点（只有实体节点，把原文存在实体属性里）？

因为一段原文可能包含多个实体。如果把原文存在实体属性里，同一段文本会被重复存储多次。双节点结构是多对多关系的自然建模：多个实体可以指向同一个 PassageNode。

### Q2：MENTIONED_IN 最多 3 条，会不会漏掉重要证据？

3 条是按匹配分数排序后取的 top-3，最相关的段落一定在里面。如果一个实体在 10 个段落中都被提到，说明它是高频实体，取 top-3 已经足够覆盖核心语境。剩余段落通过向量检索仍然可达。

### Q3：RELATED_TO 只有 6 种关系类型，会不会太少？

这 6 种是医疗领域最高频的关系类型，覆盖了绝大多数临床问答场景。如果扩展到通用关系（如"属于"、"位于"），会引入大量低价值边，增加遍历噪音。宁可少而准，不要多而杂。

### Q4：句子级模式推断的准确率如何？

比 LLM 抽取低，但成本为零。它的定位是"补充"而非"替代"——LLM 抽取覆盖骨架 chunk，句子级推断覆盖外围 chunk 中明显的关系模式。两者互补。

### Q5：HippoRAG 2 的原始方案和你的实现有什么区别？

HippoRAG 2 用 Personalized PageRank（PPR）做查询时的动态排序。我的实现保留了 PPR（`compute_ppr()` 函数），但主要用在 VectorCypher 的遍历阶段——从 query 相关的入口节点出发，用 PPR 对遍历到的节点做相关性排序。另外我加了医疗领域的实体类型约束和关系类型约束，这是论文没有的。

### Q6：如果两个实体之间有多种关系（比如 ACEI 对高钾血症既是"禁忌"又是"副作用"），怎么处理？

会建两条 RELATED_TO 边，relation_type 不同。Cypher 遍历时可以按 relation_type 过滤，只返回用户关心的关系类型。
