# 检索自纠循环

## 简历原文

> 基于 LangGraph 构建双层状态图，内层通过 LLM 评估召回质量并输出结构化输出，驱动查询改写或工具切换；外层在生成后通过 CoVe 对答案事实声明逐条图谱验证，证据不足时自动触发补充召回。

---

## 一、为什么需要自纠循环？

传统 RAG 是"检索一次 → 生成答案"的单次流程。问题：

1. **检索可能失败**：路由选错了工具、query 表述不好、索引中没有直接匹配
2. **无法感知失败**：系统不知道检索结果质量好不好，直接把垃圾结果喂给 LLM
3. **LLM 幻觉**：基于低质量证据生成的答案可能包含事实错误

自纠循环的核心思想：**检索后先评估质量，质量不够就换策略重试，生成后再验证答案的事实准确性**。

---

## 二、双层状态图架构

整个系统用 LangGraph（StateGraph）实现，分为两层：

### 外层：Agent Workflow（端到端流程）

```
route_query → retrieve_evidence → generate_answer → verify_answer → check_completeness → augment
```

6 个节点：
1. **route_query**：分类 query type，选择初始工具
2. **retrieve_evidence**：调用内层自纠循环，拿到最终检索结果
3. **generate_answer**：用检索结果 + LLM 生成答案
4. **verify_answer**：CoVe 声明验证（下面详细讲）
5. **check_completeness**：全局查询的完整性检查
6. **augment_with_comprehensive**：证据不足时补充召回

### 内层：Self-Correction Loop（检索质量纠错）

```
execute_attempt → evaluate_reflection → interpret_verdict → rerank/prepare_retry/finish
```

5 个节点：
1. **execute_attempt**：执行当前工具，缓存结果
2. **evaluate_reflection**：LLM Judge 评估召回质量
3. **interpret_verdict**：解析评估结果，决定下一步
4. **rerank_results**：对当前结果重新排序
5. **prepare_retry**：选择下一个工具 + 可选查询改写

---

## 三、内层详解：Self-Correction Loop

### 执行（execute_attempt）

运行当前工具（如 vector_search），记录：
- 返回结果数量
- 执行耗时
- 哪些通道是复用缓存的、哪些是新执行的

### 评估（evaluate_reflection）

**跳过优化**：如果 vector 通道的 top-1 结果 cosine similarity ≥ 0.85，直接跳过 LLM Judge，认为结果足够好。这避免了对高质量结果的无谓 LLM 调用。

**LLM Judge 评估**：对于不够好的结果，调 LLM 评估，输出结构化 ReflectionStep：

| 字段 | 含义 |
|------|------|
| evidence_status | sufficient / partial / insufficient / empty |
| gap_type | missing_entity / missing_relation / off_topic / none |
| verdict | answer / rerank / retry / stop |
| failure_type | relation_missing / insufficient_context / no_results / ... |
| missing_entities | 缺失的实体列表 |
| missing_relationships | 缺失的关系列表 |
| recommended_action | 推荐的下一步动作 |
| should_rewrite_query | 是否需要改写查询 |
| preferred_tools | 推荐的工具列表 |
| preferred_providers | 推荐的通道列表 |

### 判决解释（interpret_verdict）

根据 verdict 分流：

- **answer**：证据充分，结束循环，返回结果
- **rerank**：证据在但排序不好，重新排序后再评估
- **retry**：证据不足，切换工具重试
- **stop**：预算耗尽或无法改善，返回当前最好结果

### 重试准备（prepare_retry）

1. **选择下一个工具**：
   - 优先用 reflection 推荐的工具
   - 其次查 retry matrix
   - 跳过已经试过的工具

2. **增量 hybrid retry**：
   - 如果 reflection 说"graph 通道缺关系"，只重跑 graph 通道
   - 复用已缓存的 vector 和 bm25 结果
   - 重新融合

3. **查询改写**（可选）：
   - 当切换到 comprehensive_search 或 full_document_read 时触发
   - LLM 根据 reflection 的 missing_entities 生成新查询
   - 防止重复：改写后的查询不能和历史查询重复

### 循环守卫

防止无限循环的机制：

1. **最大重试次数**：默认 1 次（可配置）
2. **时间预算**：默认 1500ms，超时强制停止
3. **重复 gap 检测**：如果 reflection 连续两次报告相同的 missing claims，停止
4. **已覆盖 gap 检测**：如果 reflection 报告的 missing entity 实际已经在结果中出现，停止（说明 LLM Judge 幻觉了）

## 四、外层详解：CoVe 声明验证

### 什么是 CoVe？

Chain-of-Verification（CoVe）：把生成的答案拆成原子事实声明，逐条验证每个声明是否有证据支撑。

### 触发条件

不是所有答案都需要验证。触发条件（满足任一）：
1. query type 是 relation / multi_hop / global
2. 答案包含多个可验证事实（≥2 个分句 + 数字/剂量/阈值）
3. 检索过程中用了 graph/hybrid 工具

### 验证流程

1. **声明提取**：LLM 把答案拆成原子事实声明
   - 输入："ACEI 禁用于高钾血症患者，可改用 ARB"
   - 输出：["ACEI 禁用于高钾血症", "高钾血症患者可改用 ARB"]

2. **逐条验证**：对每个声明，在图谱中查找支撑证据（通过 cypher_traverse，不调 LLM）
   - 查 PhraseNode("ACEI") --RELATED_TO{type:"禁忌"}--> PhraseNode("高钾血症") 是否存在

3. **标记结果**：
   - 所有声明有支撑 → answer_status = "verified"
   - 部分声明无支撑 → answer_status = "partial"
   - 需要重试 → answer_status = "retry_required"

### 完整性检查（全局查询专用）

对于 global 类型查询（如"列出所有降压药分类"），生成答案后额外检查：
- 如果 retrieval_status 是 partial/empty/timeout
- 或者最后一次 reflection 的 verdict 不是 answer
- → 触发 comprehensive_search 补充召回，重新生成答案

## 五、状态追踪与可观测性

整个流程产生 PipelineTrace，记录每一步的决策和数据：

| 组件 | 记录内容 |
|------|---------|
| RouterStep | 路由方法、分类结果、耗时 |
| ToolStep | 工具名、结果数量、耗时、缓存命中、通道诊断 |
| ReflectionStep | verdict、evidence_status、missing claims |
| EscalationStep | 从哪个工具升级到哪个、原因、改写后的查询 |
| GeneratorStep | 模型名、token 数、answer_status |
| WorkflowMemory | 每个阶段的结构化日志 |

## 六、面试追问

### Q1：LLM Judge 评估一次要多久？会不会太慢？

单次 LLM Judge 调用约 200-500ms。通过跳过优化（cosine ≥ 0.85 直接跳过）和时间预算（1500ms 总预算），大部分简单查询不会触发 Judge。只有检索质量确实不好时才调。

### Q2：自纠循环最多重试几次？

默认 1 次（`AGENT_MAX_RETRIES=1`）。加上初始执行，最多执行 2 次检索。这是延迟和质量的平衡——重试太多次用户等不了。

### Q3：如果 LLM Judge 本身判断错误怎么办？

两个防护：
1. **循环守卫**：如果 Judge 说"缺少实体 X"但 X 已经在结果中，直接停止（检测到 Judge 幻觉）
2. **best_results 追踪**：始终保留历史最好的结果集。即使 Judge 误判导致 retry 后结果变差，最终返回的是历史最好的那一批

### Q4：CoVe 验证失败后会怎样？

当前实现中，CoVe 只标记 answer_status（verified/partial），不会触发重新生成。这是设计选择——CoVe 的目的是让用户知道答案的可信度，而不是无限循环重试。

### Q5：LangGraph 相比直接写 while 循环有什么优势？

1. **状态管理**：StateGraph 自动管理复杂状态（20+ 个字段），不需要手动传参
2. **条件分支**：conditional_edges 让分支逻辑声明式表达，比 if-else 嵌套清晰
3. **可观测性**：每个节点的输入输出自动记录，方便调试
4. **编译优化**：`@lru_cache` 编译一次，后续请求复用

### Q6：时间预算 1500ms 够吗？

对于单次检索 + 跳过 Judge 的简单查询，绰绰有余（通常 200-400ms）。对于需要 retry 的复杂查询，1500ms 大约够一次 retry（检索 200ms + Judge 400ms + 第二次检索 200ms）。如果预算不够，循环守卫会强制停止并返回当前最好结果。

### Q7：增量通道刷新和全量重跑相比，性能提升多少？

假设三个通道各 200ms，全量重跑 = 600ms（并行也要 200ms + 开销）。增量刷新只跑一个通道 = 200ms，复用两个缓存通道 = 0ms。节省约 2/3 的检索时间。

### Q8：这个系统和 LangChain 的 Self-RAG 有什么区别？

Self-RAG 是在生成阶段做 token 级别的自我反思（每生成一段就判断是否需要检索）。我的方案是在检索阶段做 document 级别的质量评估（检索完一批文档后判断够不够）。两者关注点不同：Self-RAG 关注生成质量，我关注检索质量。另外我加了 CoVe 做生成后验证，相当于两层保障。
